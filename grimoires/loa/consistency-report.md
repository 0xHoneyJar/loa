## Detailed analysis

### Skill naming (high consistency, mixed forms)

**Patterns observed**:
- **Verb-form (gerund-prefix)**: `riding-codebase`, `auditing-security`, `designing-architecture`, `discovering-requirements`, `planning-sprints`, `implementing-tasks`, `reviewing-code`, `enhancing-prompts`, `deploying-infrastructure`, `translating-for-executives`, `mounting-framework`, `bug-triaging`, `browsing-constructs`, `validating-construct-manifest`, `managing-credentials`, `continuous-learning`. **(14)**
- **Noun-noun**: `flatline-knowledge`, `agent-network-envelope`, `cost-budget-enforcer`, `hitl-jury-panel`, `loa-setup`, `red-teaming`, `rtfm-testing`, `eval-running`, `run-mode`, `run-bridge`, `simstim-workflow`, `bridgebuilder-review`, `butterfreezone-gen`, `scheduled-cycle-template`. **(14)**
- **Single-word/noun**: `spiraling`, `autonomous-agent`, `flatline-knowledge`. **(3)**

**Inconsistency**: Mixed verb-vs-noun form. The README implies "skills are named by what they do" (gerund form is preferred), but newer infrastructure skills use noun form (`hitl-jury-panel`, `cost-budget-enforcer`).

**Score**: 8/10 — kebab-case is universal, but two competing morphological patterns (gerund vs. noun-noun) reflect different naming philosophies. Not breaking, but worth a documented convention if the operator wants future skills to align.

### Command naming (universal kebab-case)

All 53 commands use kebab-case. The mapping between skills and commands is intentionally NOT 1:1 — for example:
- `discovering-requirements` skill ↔ `/plan-and-analyze` command
- `designing-architecture` skill ↔ `/architect` command
- `planning-sprints` skill ↔ `/sprint-plan` command

This is documented as "skill is the agent role; command is the user verb." The mapping is documented in README's truenames table at line 105-115.

**Score**: 10/10 — perfectly consistent format; meaningful naming separation between user-facing commands and underlying skills.

### Skill ↔ command mapping (intentionally varied)

The framework has **31 skills** and **53 commands**. The relationship:
- Skills typically map to one or more commands (e.g., `run-mode` skill backs `/run`, `/run-status`, `/run-halt`, `/run-resume`, `/run-sprint-plan`, `/run-bridge`)
- Some commands have no skill (e.g., `/contribute`, `/eval`, `/feedback`, `/oracle-analyze`)
- Some skills have no command (e.g., `flatline-reviewer`, `flatline-scorer`, `flatline-skeptic`, `gpt-reviewer` — these are sub-agents in skills under `.claude/skills/`)

This is intentional separation; `README.md:122-141` lists 18 user-facing skills (a curated subset for the public table).

**Score**: 9/10 — The intentional asymmetry is correct, but not surfaced clearly anywhere; users may be confused that the skill count claim (18) ≠ filesystem count (31).

## Cross-area conflicts

**None significant.** kebab-case is universal across files, scripts, hooks, schemas, and skills. The only naming-form variance is in skill morphology (verb-gerund vs. noun-noun) — and that's stylistic, not breaking.

## Improvement opportunities (no breaking changes)

1. **Document the skill morphology preference**. Add a one-sentence rule in CONTRIBUTING.md: "New user-facing skills SHOULD use the verb-form `<gerund>-<noun>` pattern; infrastructure skills MAY use noun-noun."

2. **Document the 18 vs 31 skills distinction**. Either:
   - Update README to enumerate all 31 skills (drift fix H1), OR
   - Add a "Public skill catalog vs. internal-skills" footnote in the table.

3. **Document the skill ↔ command mapping**. Adjacent table or column showing `/command` ⇄ `<skill-name>` for the 18 public skills.

4. **Consider organizing `.claude/scripts/` by category like hooks**. Currently 277 top-level scripts could be grouped under `audit/`, `beads/`, `budget/`, `git-hooks/`, `lib/`, `tests/`, `trace_analyzer/` consistently — already partially done. Reduces cognitive load and matches `.claude/hooks/` precedent.

## Final score: 9/10

Naming consistency is exceptionally high. The single point lost reflects the verb-vs-noun ambiguity in skill names, which is stylistic and would require explicit operator decision before any rename. No code action recommended.
