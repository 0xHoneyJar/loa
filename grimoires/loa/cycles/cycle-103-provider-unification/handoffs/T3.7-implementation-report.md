## Technical Highlights

- **Karpathy-aligned scope.** One file modified, ~60 lines added (helper + normalization), one test file with 29 cases. No new abstractions beyond what the AC mandated. Reused the cycle-099 sprint-1E.c.3.c pattern (NFKC + control-byte strip) rather than inventing a new defense.
- **Single-pass percent-decode avoids amplification.** A naive `unquote` would recursively decode `%2520Bearer%2520tok` ŌåÆ `%20Bearer%20tok` ŌåÆ `Bearer tok`. The single-pass map handles the common one-level encoding without opening an amplification surface. If operators report two-level-encoded bypasses in the wild, that's a future cycle decision.
- **Zero-width charclass is comprehensive.** `[ŌĆŗ-ŌĆŹ’╗┐ŌĆ¬-ŌĆ«]` covers: zero-width space/non-joiner/joiner, BOM, left-to-right/right-to-left/pop directional formatting/embedding/override. The four zero-width characters tested individually represent the dominant insertion-attack vectors.
- **`[REDACTED-Bearer]` sentinel safe.** The post-redactor sentinel lacks the `{16,}` alphanumeric run after `Bearer`. Verified by `test_redacted_sentinel_does_not_trip`.
- **Polar bears safe.** Negative control: text mentioning the word "bear" (animal noun) doesn't trip. The regex's word-boundary-via-charclass-context (`[Bb]earer` followed by separator + token) is unambiguous for plausible English text.

## Known Limitations

1. **Two-level percent-encoding not handled.** `%2520Bearer%2520tok` (URL-encoded percent sign) would NOT be decoded. The single-pass design is intentional (amplification defense). If operator monitoring surfaces this vector, extend to bounded-depth multi-pass.
2. **HTML entity encoding not handled.** `&#x20;Bearer&#x20;tok` (HTML decimal/hex entities) won't decode. Same disposition ŌĆö not in the documented threat model from DISS-004. Add when needed.
3. **Base64 / hex-encoded bearer not detected.** A bearer token embedded as a base64 blob wouldn't be recognized as a bearer shape. That's a different threat model (the redactor would treat it as opaque bytes); not in scope for T3.7.
4. **Pattern still requires literal `Bearer` / `bearer` prefix.** Variants like `Token`, `API-Key`, `X-Auth` aren't covered. The gate is intentionally narrow on Bearer because that's the highest-leakage shape for upstream-bytes-to-stderr per PRD ┬¦5.2. Adding more shapes is straightforward when needed.
