## Known Limitations

1. **No CI gate keeps the runbook in sync with code.** If a future cycle
   changes the env-var contract (e.g., renames `LOA_BB_FORCE_LEGACY_FETCH`),
   the runbook silently drifts. Mitigation: §10 version-stamps the runbook
   to cycle-103 sprint-1 so a future reader knows whether to trust it.
2. **The ASCII diagram is informal.** A Mermaid diagram would render in
   GitHub but be invisible in plain `cat`. Operators reading this in a
   terminal benefit from the ASCII version; operators on GitHub get the
   same content with rougher edges. Trade-off accepted.
3. **The troubleshooting matrix is non-exhaustive.** It covers the 9 most
   likely operator-encountered failures. Edge cases (cheval-side budget
   exhaustion, IPC deadlock, simultaneous-write race on
   `.run/cheval-daemon.pid`) are left to the SDD and source code. The
   "first-fix" column is intentionally conservative — wrong-but-cheap is
   acceptable for the first try.
4. **No deep guide for fixture-mode authoring.** Operators wanting to
   replay a recorded provider response into the fixture-mode contract
   should consult `T1.5-implementation-report.md` for the JSON shape
   spec. The runbook §4 shows the smoke-test fixture but not the full
   schema.
