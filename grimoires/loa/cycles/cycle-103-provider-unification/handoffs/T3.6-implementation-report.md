## Tasks Completed

### T3.6 — Path-aware nested walk

**File modified:** `.claude/adapters/loa_cheval/audit/modelinv.py`

**Structural change:** the public `redact_payload_strings` now wraps a
recursive helper `_redact_recurse(node, under_untrusted)`. The `under_untrusted`
flag enters as `False` and becomes `True` the moment a dict key in
`_REDACT_FIELDS` is traversed; once set, it propagates to every descendant
visit. At each terminal string visit, the flag decides whether `_REDACT()`
is called.

The two structural cases:

1. **Dict**: for each key `k`, compute `child_untrusted = under_untrusted or (k in _REDACT_FIELDS)`. If the value is a string AND `child_untrusted` is True, redact. Otherwise recurse with `child_untrusted`.
2. **List**: each element inherits the parent's `under_untrusted` flag. String elements directly under an untrusted list get redacted; nested dicts/lists recurse with the flag carried.

Non-string, non-dict, non-list values (int, bool, None, float) pass through
unchanged — the redactor operates on string content only, and the audit
envelope must round-trip these types byte-equal.

### Test design note: content-shape redaction semantics

The redactor (`.claude/scripts/lib/log-redactor.py::redact`) is
content-pattern based — it transforms strings containing real secret
SHAPES (Bearer tokens, AKIA keys, PEM blocks) and passes ordinary strings
through unchanged. So testing "was redact called?" via a plain string like
`"leak"` would produce false positives (redactor called, but no
transformation visible).

The tests instead embed a real Bearer-token shape
(`"Bearer abcdef0123456789xyzABCDE"`) at every test position and assert
the post-walk string contains `[REDACTED-BEARER-TOKEN]` AND no longer
contains the raw shape. This proves both (a) redact was called AND
(b) the call had the intended effect.

`TestStructurePreservation::test_plain_string_under_redact_unchanged`
pins the negative-shape case: a plain string under a redact-ancestor
still goes through the redactor, but the redactor's content-pattern
matching leaves it unchanged. Important because this is what happens to
real adapter exception strings that don't contain a secret shape — the
walk visited the value, no transformation was needed.

## Technical Highlights

- **Karpathy-aligned scope.** One file modified (~25 added lines for the helper + docstring), one test file with 22 cases. No new abstractions beyond what the AC mandated. The `under_untrusted` flag is the minimum state the recursion needs — the alternative (computing a full path tuple per visit, or building a JSON pointer) would over-engineer for no behavioral gain.
- **Round-trip preservation is the load-bearing invariant.** Sprint.md R8a pins the audit-envelope contract: after redaction, the payload must JSON-encode/decode to itself byte-equal, and applying redaction a second time must be a no-op. The `_redact_recurse` design preserves both: dict ordering follows Python 3.7+ insertion order (Python's spec guarantee, not an implementation detail), list ordering and length are explicit, non-string types are returned `is`-equal.
- **Sibling isolation is intentional, not incidental.** Tests pin that an `error_message` dict alongside a `model` dict redacts only `error_message`'s subtree. This matters because `model` / `metadata` / `params` carry operator-controlled identifiers — if the redactor were applied to them, audit-query semantics would break (one could no longer pattern-match emitted entries by their model identifier).
- **Idempotence under double application** is a critical operational property. The chain-recovery path (`audit_recover_chain`) and any log-replay flow may re-walk an emitted payload. If redaction wasn't idempotent, recovery could produce a different envelope than the original — breaking the hash chain. `TestAuditEnvelopeRoundTrip::test_idempotent_under_double_application` pins this.
- **The original bug shape (DISS-003) is realistic.** Adapter exception wrappers (especially `provider_error_message` from the Anthropic SDK and `error.detail` from the OpenAI SDK's structured-error path) frequently arrive as `{"error_message": <nested>}` rather than flat strings. Before T3.6, those nested shapes leaked the upstream body verbatim through the field gate.

## Known Limitations

1. **Tuple types not handled.** The walk recognizes `dict` and `list`. If an adapter ever passes a tuple under a redact ancestor, it would pass through unchanged. Out of scope — `json.dumps` rejects tuples upstream of the audit envelope, so this can't happen in production paths today. If a future adapter introduces tuples, extend the isinstance check.
2. **Dict-keys-with-secret-shapes not redacted.** The redactor walks VALUES, not KEYS. If a malicious adapter ever returned `{"AKIAIOSFODNN7EXAMPLE": "..."}` under a redact ancestor, the secret-shape KEY would survive. Out of scope — keys come from typed pydantic models in the adapter layer, never from upstream API response shapes.
3. **No depth limit.** A pathological adapter response with 10,000 levels of nesting could cause Python recursion-limit exhaustion. Out of scope — production adapter responses are bounded by httpx body-size limits (KF-008 mitigations) and have no nesting deeper than ~10 levels in practice. If this surfaces, switch the recursion to an explicit stack.
4. **Performance scaling.** O(n) in payload-tree size with no caching. For the 100-200-node payloads typical of adapter exceptions, this is negligible (<1ms). No optimization needed.
