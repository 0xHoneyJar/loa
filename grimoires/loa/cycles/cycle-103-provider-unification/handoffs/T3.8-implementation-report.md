## Technical Highlights

- **Karpathy-aligned: minimum-scope decision.** No code touched. No new abstractions. The deferral is a single Decision Log entry + two sprint.md edits. The whole task is ~70 lines of documentation across two files.
- **The deferral preserves optionality.** The SDD §5.2 daemon UDS protocol spec stays in the cycle-103 SDD. The TS delegate's constructor option `mode?: "spawn" | "daemon"` stays in the type. cycle-104+ can re-enable daemon-mode without breaking the cycle-103 API surface — only the code path needs to come back.
- **Structural inapplicability is the load-bearing argument.** A reader could argue "well, you COULD implement a partial connection-pool-tuning even in spawn-mode by passing tuning hints via env vars to each freshly-spawned process." That argument fails because (a) the cross-call concurrency measurement still has no shared state to read from, and (b) the per-provider connection pools can't carry across invocations. AC-3.8 isn't a small refactor — it's a daemon-mode feature that depends on long-lived state.
- **The M5 invariant has a designed-in deferral clause.** sprint.md L249 says "All 8 Sprint 4A carry-forward items closed (or AC-3.8 explicitly re-deferred with rationale)." The "or" is explicit — the cycle planning anticipated this deferral path. T3.8 is exercising the designed-in path, not a workaround.

## Known Limitations

1. **No automated test that AC-3.8 stays deferred in cycle-104.** A future cycle that "implements AC-3.8" without re-checking spawn-vs-daemon precondition would slip through. Mitigation: the Decision Log entry explicitly names the cycle-104 trigger ("spawn-per-call p95 past 1000ms"); a cycle-104 author looking at the entry would see the prerequisite.
2. **The deferral doesn't enumerate concrete cycle-104 next-steps.** It says "any workload that drives spawn-per-call p95 past 1000ms" is the trigger, but doesn't prescribe what cycle-104 should ship if that trigger fires. Out of T3.8 scope — that's a cycle-104 planning question, not a cycle-103 deferral question.
3. **No re-benchmark cadence.** Sprint 1 R1 measured p95=126ms on a specific benchmark shape (50 sequential + 3 concurrent calls per provider). If the BB review pass shape changes (e.g., more concurrent calls, larger prompts), the spawn-mode margin could shrink. The Decision Log entry implicitly accepts this; a periodic re-benchmark would catch drift. Not blocking cycle-103.
