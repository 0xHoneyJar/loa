## Tasks Completed

### T1.11 — Bats regression-catcher for the escape-hatch trip

**File created:** `tests/unit/cheval-delegate-legacy-fetch-trip.bats` (~110 lines, 8 tests)

**Test contract (the 8 pinned clauses):**

| # | Test | What it pins |
|---|------|--------------|
| 1 | env-var name is referenced | Fails if `LOA_BB_FORCE_LEGACY_FETCH` is renamed without coordinating with the runbook + TS test |
| 2 | uses `=== "1"` strict equality | Fails if someone weakens to truthy check (`if (env)`) — which would trip on `"false"`, `"0"`, etc. |
| 3 | throws `LLMProviderError(INVALID_REQUEST, ...)` | Pins the typed-error contract per SDD §5.3 |
| 4 | error message references `cheval-delegate-architecture.md` | Operators must find their way to T1.10's runbook |
| 5 | error message mentions `cycle-103` | Anchors the cycle ID for stale-clone debugging |
| 6 | trip happens in the constructor (fail-fast) | Pins location-in-source — env check must precede side effects |
| 7 | runbook cross-link present | Asserts T1.10 runbook references the env var name |
| 8 | TS-level contract test still present | Asserts `cheval-delegate.test.ts` still has test #3 — belt-and-suspenders against silent TS-test removal |

**Why grep-based, not behavioral:** the TS test (`cheval-delegate.test.ts:127-148`) already pins the **behavior**. This bats file pins the **source text**. A refactor that changes the implementation while keeping the behavior would update both — but a refactor that *accidentally* drops the contract (e.g., a careless cleanup PR that removes the trip block thinking it's dead code) would now fail in TWO independent test suites. The cost is low (~110 lines of bats) and the safety is real.

### Why I did NOT add a behavioral bats test

A behavioral bats test for the constructor trip would need to spawn the BB skill end-to-end with `LOA_BB_FORCE_LEGACY_FETCH=1` in env and capture the error. That requires:
1. A TS test runner accessible from bats (the existing `npx tsx --test` works but spawns Node).
2. An entry point that constructs the adapter without actually calling a provider.

Option (1) is already covered by the TS test (`cheval-delegate.test.ts` test #3 inside the suite). Option (2) doesn't exist as a standalone shell-callable surface. Adding either would duplicate the TS coverage at higher cost. The grep-pins are cheaper AND catch a strictly larger class of regressions (source-text removal).

## Technical Highlights

- **Karpathy-aligned: test-only, no production code change.** T1.11 is a 110-line bats file. The actual escape-hatch behavior was implemented in T1.2; T1.11 just hardens its survivability across future refactors.
- **Belt-and-suspenders with the TS test.** Removing the escape hatch silently now requires removing BOTH the TS-level behavior test (1 test in `cheval-delegate.test.ts`) AND all 8 source-text pins in the new bats file. Both removals would be visible in PR diff. A reviewer looking at a PR that touches `cheval-delegate.ts` would see at least one test deletion — that's the load-bearing visibility.
- **Test #6 pins the fail-fast invariant.** The trip MUST happen in the constructor, not lazily in `generateReview()`. If a refactor moved the check to lazy evaluation, an adapter-factory call site that constructs the adapter without immediately calling `generateReview()` would silently succeed — postponing the error until much later. The line-position assertion catches this.
- **Test #8 nests the TS test into the bats contract.** This is unusual: a bats test that asserts a TS test file exists with a specific test name. It works because the TS test is grepped as source text, not executed. The bats file is now load-bearing for the TS test's *presence*, not its execution — separating concerns of "test still exists" from "test still passes".

## Known Limitations

1. **Grep-based pins are brittle to syntax changes.** If `cheval-delegate.ts` is reformatted (e.g., the constructor body is split across multiple lines, the comparison is broken into `const flag = process.env.LOA_BB_FORCE_LEGACY_FETCH; if (flag === "1")`), some tests may fail spuriously. Mitigation: each test's failure produces a clear error pointing to the specific contract clause; the fix is to update the bats test alongside the implementation change.
2. **No coverage for non-`"1"` values that should NOT trip.** Test #2 asserts the comparison is `=== "1"`, which implicitly excludes `"true"` / `"yes"` / `"on"` etc. — but doesn't test that those values DON'T trip. The TS-level test could add this; out of T1.11 scope.
3. **Test #8 creates a one-directional dependency.** If someone wants to rename the TS test description (test #3's name "LOA_BB_FORCE_LEGACY_FETCH=1 triggers guided rollback error"), they must also update test #8 here. The rename is acceptable; the silent drop is the failure mode this guards.
