3. **`dispatch_provider_stream_error(error, provider="")` function** (`types.py:397-455`) â€” single lookup table from `ProviderStreamCategory` â†’ `ChevalError` subclass:

   | Category | Dispatched type | Retry semantics |
   |----------|-----------------|-----------------|
   | rate_limit | `RateLimitError` | retry with backoff |
   | overloaded | `ProviderUnavailableError` | move to next provider |
   | malformed | `InvalidInputError` | surface to caller |
   | policy | `InvalidInputError` | surface to caller |
   | transient | `ConnectionLostError` (issue #774 sibling) | per-provider retry budget |
   | unknown | `ProviderUnavailableError` | conservative fallback |

   Empty `provider=""` defaults to `"unknown"` in the constructed exception's context (so callers that don't know the provider still produce well-formed exceptions).

## Technical Highlights

- **R4 mitigation satisfied.** sprint.md L212 mandates "T3.1 lands FIRST as foundational; T3.2â€“T3.7 layer on top." This commit ships ONLY the typed exception + dispatch table. No adapter code is rewritten; retry.py is untouched. T3.5 (SSE buffer caps) is the first task that will actually raise `ProviderStreamError` from a streaming parser; T3.5 catches + dispatches at the adapter layer.
- **Single lookup table â€” no cascade.** The dispatch is one function with an if/elif chain. retry.py reads only the dispatched typed exception (which is one of the existing exception classes), so retry.py is structurally identical to its pre-T3.1 form. PRD R4 specifically warned against "cascading rewrite through retry.py + cheval.py + all 3 adapter wrappers"; the lookup-table approach avoids this entirely.
- **Sibling, not subclass.** `ProviderStreamError` extends `ChevalError` directly, NOT `ProviderUnavailableError` / `RateLimitError`. Reason: subclassing would mean `except RateLimitError:` catches it, which would dispatch the wrong retry semantics for non-rate-limit categories. By making it a sibling + requiring explicit dispatch, the type system forces callers to convert before retry.py sees it.
- **`raw_payload` redaction is caller-side.** Per T3.3 sequencing (`sanitize_provider_error_message` ships in T3.3), the `raw_payload` field is `Optional[bytes]` and the docstring explicitly notes "Caller is responsible for redaction before persisting or surfacing." T3.3 will add the helper; until then, callers should NOT log `raw_payload` directly. The field exists so it CAN be redacted-and-logged in T3.3+ without re-raising.
- **`retryable` consistency check.** The construction-time `retryable` flag on `ProviderStreamError` matches what `dispatch_provider_stream_error` returns. Two tests pin this from different angles (`test_retryable_mirrors_dispatched_semantics` parametrize Ă— `test_dispatch_preserves_retry_semantics_per_AC_3_1`). Single subtle wart: `unknown` is `retryable=False` on the raw exception but `retryable=True` after dispatch (because the conservative fallback is ProviderUnavailableError, which IS retryable). Documented in code comment + the parametrize in `TestRetrySemantics`.

## Known Limitations

1. **No streaming parser raises `ProviderStreamError` yet.** This is intentional â€” T3.1 ships the type + dispatch. T3.5 (SSE buffer caps) is the first task that raises it from `anthropic_streaming.py` / `openai_streaming.py` / `google_streaming.py`. The pattern documented in the docstring + the round-trip test shows the intended usage.
2. **`sanitize_provider_error_message` not wired into `raw_payload`.** Per T3.3 sequencing â€” the redactor ships separately. Until T3.3 lands, callers MUST manually redact `raw_payload` before logging. The risk is mitigated because no caller currently consumes `raw_payload` (no T3.1 caller exists).
3. **`retry.py` untouched.** Pre-existing retry.py expects ChevalError subclasses with `retryable: bool`. Every dispatched exception in our table has the right `retryable` semantics, so retry.py reads them transparently. Verified by the `test_dispatch_preserves_retry_semantics_per_AC_3_1` test.
4. **`unknown` â†’ `ProviderUnavailableError`** is the conservative choice. Could argue `unknown` should raise to caller (surface unclassified errors). Going with PU because the alternative (surfacing) would block retry-chain progression on a single weird event from one provider. If operator monitoring surfaces unexpected `unknown` rates, the streaming parsers should be extended to classify more precisely â€” not the dispatch logic.
