## Technical Highlights

- **Karpathy-aligned scope.** T1.4 is a surgical migration: 3 files modified, 6 deleted, no new abstractions added. The legacy adapter registry pattern was useful when each provider lived in its own file with its own HTTP loop; with cheval consolidating that surface, the registry is dead weight.
- **Caller compat preserved.** `multi-model-pipeline.ts:170` still calls `createAdapter({ provider, modelId, apiKey, timeoutMs, costRates })` exactly as before. No caller-side changes required. `apiKey` and `costRates` are accepted-and-ignored. The diff is invisible to upstream consumers ŌĆö the BEHAVIOR changes (calls now route through Python) but the API surface is identical.
- **Net-loss tests is intentional.** The 55-test reduction (pre-T1.4 749 / post-T1.4 694) is correct. The legacy per-provider tests were structural-shape tests against TS adapters that no longer exist. The shape-validation they covered (Anthropic request format, OpenAI message routing, Google SSE parsing) has moved to cheval-Python's own pytest suite ŌĆö that's where the live provider HTTP loop now lives. Trying to keep TS-side mirror tests for retired adapters would have been dead coverage.
- **No `registerAdapter` survivors.** The registry concept is gone. If a future cycle needs runtime provider injection, the right hook is `ChevalDelegateAdapter`'s `spawnFn` constructor option (used in tests) or `chevalScript` override ŌĆö both already exist from T1.2.
- **One commit, no rollback drama.** All 9 file operations (3 edits + 6 deletes) are atomic with the T1.4 commit. `git revert <T1.4 commit>` restores the full legacy registry + per-provider adapters + their tests.

## Known Limitations

1. **`AnthropicAdapter` direct-import from outside BB will break.** Any external consumer that imported `AnthropicAdapter` / `OpenAIAdapter` / `GoogleAdapter` from `@loa/bridgebuilder-review/adapters` is now broken. A repo-wide grep returned zero such consumers, but external tooling outside this repo cannot be checked from here. Mitigation: AC-1.5 explicitly says these stay in TS ŌĆö meaning the **wrapper paths** (`github-cli.ts`, `.reviewignore`, etc.) stay. The provider-level adapters going away is the explicit cycle-103 deliverable. Operators relying on direct imports should pivot to `ChevalDelegateAdapter`.
2. **`registerAdapter` / `getRegisteredProviders` are gone.** Any caller relying on these for runtime provider injection (e.g., mock provider injection in legacy tests) will break. Repo-wide grep returned zero non-test consumers; the only call site was `__tests__/adapter-factory.test.ts` itself, which was rewritten.
3. **Single-model BB still pre-checks `ANTHROPIC_API_KEY`.** `createLocalAdapters` throws if `anthropicApiKey === ""`. This is intentional fail-fast behavior for the single-model default ŌĆö the value itself is not used by the delegate (env-inheritance), but its presence in the env is a precondition. Operators who want a different default provider for single-model BB can change `main.ts:471` to check a different env var. Out of T1.4 scope.
4. **The agent name is hardcoded to the delegate's default ("reviewing-code").** `adapter-factory.ts:46-53` does not pass an `agent` field to the `ChevalDelegateAdapter` constructor, so it falls through to the delegate's default. For BB cross-model reviews, the same agent binding routes all three providers ŌĆö fine for current scope, but a future cycle may want per-provider agents (e.g., `jam-reviewer-claude` / `jam-reviewer-gpt`) so cheval-side personas can vary by model. Documented as `AdapterConfig.agent` for that future cycle.
5. **SDD ┬¦2.1 still lists the three legacy adapters under "Removed in cycle-103" but did NOT track the per-provider __tests__ deletions.** Flagged for the Sprint 1 SDD-correction sweep (alongside T1.2's `--timeout-seconds` / `--json-out` divergence and T1.5's argument-shape notes).
