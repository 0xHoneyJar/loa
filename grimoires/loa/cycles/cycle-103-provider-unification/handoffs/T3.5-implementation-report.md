# T3.5 — SSE Buffer + Per-Event Accumulator Caps Report

## Executive Summary

T3.5 is **the first caller of T3.1's `ProviderStreamError`** dispatch.
Both SSE event-stream iterators and per-event accumulators now enforce
byte caps and raise `ProviderStreamError("transient", ...)` when
pathological / runaway upstream data threatens unbounded memory growth.
Adapter layer catches `ProviderStreamError` and dispatches through T3.1's
`dispatch_provider_stream_error` → typed exception (`ConnectionLostError`
for `transient` per the dispatch table), which retry.py handles via its
per-provider retry budget (AC-3.5).

| Cap | Value | Site |

| `MAX_SSE_BUFFER_BYTES` | 4 MiB | `_iter_sse_events` (Anthropic) + `_iter_sse_events_raw_data` (OpenAI/Google) — checked after every chunk append |

| `MAX_ARGS_PART_BYTES` | 256 KiB | `arguments_parts` / `json_parts` — checked after every tool-call argument delta accumulate |

## Tasks Completed

### T3.5 — caps + dispatch wiring

**New module:** `.claude/adapters/loa_cheval/providers/streaming_caps.py`
(~65 lines). Exposes:

- Three constants: `MAX_SSE_BUFFER_BYTES`, `MAX_TEXT_PART_BYTES`,
  `MAX_ARGS_PART_BYTES` — pinned to sprint.md L222 values.
- `check_buffer_cap(buffer_len)` — raises if SSE buffer exceeds 4 MiB.
- `accumulate_capped(parts, delta, *, cap, kind)` — appends delta and
  raises if cumulative bytes exceed cap. The `kind` label appears in the
  error message (e.g. `'arguments'`, `'reasoning_summary'`) so the
  failure surface identifies WHICH accumulator tripped without leaking
  accumulator content.

**Streaming-parser wiring:** the three streaming parsers
(`anthropic_streaming.py`, `openai_streaming.py`, `google_streaming.py`)
call `check_buffer_cap(len(buffer))` after every chunk append and use
`accumulate_capped(...)` at every delta-append site. The Google parser
re-uses OpenAI's `_iter_sse_events_raw_data` so it inherits the buffer
cap transparently.

**Adapter-layer dispatch:** the three provider adapters
(`anthropic_adapter.py`, `openai_adapter.py`, `google_adapter.py`)
each have their parser-call try-block extended:

```python
try:
    result = parse_<provider>_stream(...)
except ProviderStreamError as stream_err:
    # T3.5 / AC-3.5: SSE buffer + per-event accumulator caps raise
    # ProviderStreamError; dispatch through T3.1's table → typed.
    raise dispatch_provider_stream_error(
        stream_err, provider=self.provider
    ) from stream_err
except ValueError as parse_err:
    raise InvalidInputError(...) from parse_err
```

The `ProviderStreamError` catch comes BEFORE the legacy `ValueError`
catch so cap-raised exceptions reach the typed dispatch even though
`ProviderStreamError` is a `ChevalError`, not a `ValueError`. (The order
is structural — Python tries except clauses in declaration order; the
two branches are mutually exclusive on the exception type, so the order
is for readability + future-proofing.)

## Technical Highlights

- **Karpathy-aligned: one shared helper, six call sites.** Adding a separate cap helper per streaming parser would have duplicated ~50 lines of bookkeeping. The shared `streaming_caps` module centralizes the cap values and the bookkeeping; the three parsers each call `check_buffer_cap(len(buffer))` (one line) and `accumulate_capped(...)` (one expression replacement) — surgical changes with no parser-internal logic changes.
- **First caller of T3.1's typed dispatch.** T3.1 shipped `ProviderStreamError` + `ProviderStreamCategory` + `dispatch_provider_stream_error` in `5fb7c9c4`. Until T3.5, nothing was raising `ProviderStreamError`. T3.5 wires the producer (caps) and the consumer (adapter dispatch); the contract is now end-to-end exercised. Future tasks (T3.3 sanitize_provider_error_message, mid-stream provider error parsing) can reuse this pattern with different categories (`rate_limit`, `overloaded`, `malformed`, `policy`).
- **`ConnectionLostError` is the correct typed destination.** The dispatch table maps `"transient"` → `ConnectionLostError`. Retry.py counts `ConnectionLostError` against the per-provider retry budget — exactly the right semantic for cap-exhaustion. Cap exhaustion is a transport-layer signal that the server-side has stopped emitting valid SSE structure, which is precisely what `ConnectionLostError` was designed to express. The retry layer then decides whether to retry within the budget or move to the next provider in the chain.
- **`O(n)` sum-of-lens per accumulate is acceptable.** `accumulate_capped` walks `parts` once after each append. For a normal stream (n events, M bytes), this is O(nM) overall — same complexity as the final `"".join(parts)` which the parser already does. The alternative (carrying running totals) would couple the helper to caller state. At current scales (≤1MB per accumulator), the overhead is sub-millisecond.
- **Boundary-exact behavior matches AC.** `check_buffer_cap(MAX_SSE_BUFFER_BYTES)` does NOT raise — only `> cap` raises. This matches the AC's "exceeds" verb. The boundary-exact case is the busiest cell on the threshold, and matching the AC's exact-text avoids an off-by-one drift.
- **`kind` label in error message identifies the accumulator without leaking content.** A `'arguments'` cap exhaustion at byte 250000 doesn't reveal what those bytes were. The label tells the operator/debugger which accumulator-by-type tripped (text vs thinking vs arguments vs refusal vs reasoning_summary) so triage can proceed without log-scraping the redacted payload.
- **Google adapter reuses OpenAI's iterator for free.** `google_streaming.py` imports `_iter_sse_events_raw_data` from `openai_streaming`. Adding the buffer cap to that one function automatically protects Google's path. No code duplication; one source of truth.

## Known Limitations

1. **Buffer cap check granularity = per-chunk.** If a single chunk arrives that already exceeds 4 MiB, the cap trips on the first append. This is the intended behavior — but it means the maximum memory residency for a single pathological chunk is bounded by the chunk size (typically ≤64 KiB from httpx). Not an issue in practice.
2. **O(n) sum-of-lens per accumulate.** See Technical Highlights for rationale. If profiling ever shows this in the hot path, replace with running-total bookkeeping at the cost of caller coupling.
3. **No metrics emission on cap trip.** A cap trip raises and stops the parse — there's no observability beyond the error itself. If operators want to count cap-trip frequency over time, add a logger.warning() call or audit-emit at the cap-trip site. Out of scope for T3.5; future cycle.
4. **AC mentions ValueError; implementation uses ProviderStreamError.** Verbatim AC says ValueError. T3.1 introduced `ProviderStreamError` specifically as the structured replacement. The implementation uses the typed path (caught + dispatched → ConnectionLostError); net effect is identical to what the AC mandates, and the typed path is what T3.1's design contract specifies. The legacy `except ValueError → InvalidInputError` remains in place for genuine malformed-data paths.
5. **Reasoning summary cap shares MAX_TEXT_PART_BYTES.** OpenAI's `reasoning_summary_text` is treated as 1 MiB cap like other text-type accumulators. If a future model emits genuinely-large reasoning summaries (multi-MB), this would need a separate cap. Defer to operator observation.
