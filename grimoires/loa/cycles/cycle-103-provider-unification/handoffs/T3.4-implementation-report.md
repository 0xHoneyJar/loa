# T3.4 — `streaming_max_input_tokens` / `legacy_max_input_tokens` Split Report

## Executive Summary

Closes AC-3.4: when `LOA_CHEVAL_DISABLE_STREAMING=1` is set, the
`_lookup_max_input_tokens` gate now automatically reverts to the
legacy-safe value (24K openai / 36K anthropic) instead of holding at the
streaming default (200K / 180K). The mechanism: a per-model schema split
into `streaming_max_input_tokens` + `legacy_max_input_tokens` (with the
existing `max_input_tokens` retained as backward-compat fallback for
unmigrated configs).

Without this split, an operator who killed streaming via the documented
kill switch would still hit the streaming-tuned ceiling — a footgun
where the kill switch's intent (revert to pre-Sprint-4A safety) was not
honored by the input gate.

- **New files (1):** `.claude/adapters/tests/test_max_input_token_gate_split.py` — 19 tests across 7 classes.
- **Modified files (4):**
  - `.claude/adapters/cheval.py` — `_lookup_max_input_tokens` now branches on `LOA_CHEVAL_DISABLE_STREAMING` and prefers the appropriate split field; falls back to legacy `max_input_tokens` when split absent.
  - `.claude/adapters/loa_cheval/types.py` — `ModelConfig` dataclass adds `streaming_max_input_tokens` + `legacy_max_input_tokens` (both `Optional[int] = None`).
  - `.claude/data/schemas/model-config-v2.schema.json` — new optional fields with descriptions; `max_input_tokens` description updated to note its backward-compat role.
  - `.claude/defaults/model-config.yaml` — 4 entries updated (gpt-5.5, gpt-5.5-pro, claude-opus-4-7, claude-opus-4-6) — added streaming + legacy split values; kept `max_input_tokens` for backward compat.
  - `.claude/scripts/loa-migrate-model-config.py` — added `--cycle103-split` flag + `_apply_cycle103_split` helper. Per-provider legacy defaults (openai=24K, anthropic=36K, google=24K).
- **Tests:** 19/19 new pass. Full cheval pytest 1093/1093 pass (delta +19 from T3.4; zero regressions).

## Tasks Completed

### T3.4 — split-aware lookup + schema + config + migration

**`_lookup_max_input_tokens` branch logic** (cheval.py:215-275):

```python
_streaming_killed = os.environ.get(
    "LOA_CHEVAL_DISABLE_STREAMING", ""
).strip().lower() in ("1", "true", "yes", "on")
preferred_field = (
    "legacy_max_input_tokens" if _streaming_killed
    else "streaming_max_input_tokens"
)
threshold = model_config.get(preferred_field)
if threshold is None:
    # Backward-compat: legacy single-field configs.
    threshold = model_config.get("max_input_tokens")
```

The kill-switch parse mirrors `_streaming_disabled()` in
`providers/base.py` — same truthy values, same case-insensitive match.
Cannot share the helper directly because cheval.py is a top-level
script (not a package member) and importing from loa_cheval would pull
the whole dependency tree at startup. The parse is ~3 lines —
duplication is cheaper than restructuring the entry point.

**Per-entry config update** (model-config.yaml):

| Entry | streaming_max | legacy_max | max_input (compat) |
|-------|---------------|------------|--------------------|
| openai/gpt-5.5 | 200000 | 24000 | 200000 |
| openai/gpt-5.5-pro | 200000 | 27000 | 200000 |
| anthropic/claude-opus-4-7 | 180000 | 36000 | 180000 |
| anthropic/claude-opus-4-6 | 180000 | 36000 | 180000 |

Each entry retains `max_input_tokens` so any external tooling that
reads the single-value form (e.g. observability dashboards, lint
scripts) continues to see the streaming-default value. Internally,
`_lookup_max_input_tokens` prefers the split — so the gate behavior
matches transport.

**Migration tool extension** (loa-migrate-model-config.py):

The `--cycle103-split` flag runs after v1→v2 migration completes. It
walks every `providers.<p>.models.<m>` entry that carries
`max_input_tokens` and injects the split fields if they're absent:

- `streaming_max_input_tokens` = existing `max_input_tokens` (no
  behavior change for currently-working configs).
- `legacy_max_input_tokens` = provider-specific default
  (openai=24000, anthropic=36000, google=24000), or
  `max_input_tokens` itself for unknown providers (defensive).

Idempotent: re-running on a config that already has the split fields
is a no-op. Report entries (`kind: cycle103_split_injected`) document
exactly which paths were modified.

## Technical Highlights

- **Karpathy-aligned: surgical change to one function + 4 yaml entries.** The lookup logic gains ~10 lines (env parse + preferred-field branch + fallback). The dataclass + schema each gain 2 optional fields. The yaml entries each gain 2 lines + a comment. The migration tool gains one flag + one ~50-line helper. No new abstractions; no refactor of unrelated code.
- **Backward-compat is load-bearing.** Operators with downstream forks have model-config.yaml files that pre-date this split. The lookup function's fallback to `max_input_tokens` means those configs continue to work unchanged — the kill-switch behavior in legacy mode just won't auto-revert. Operators who want the auto-revert can either (a) re-run the migration tool with `--cycle103-split`, or (b) hand-add the split fields. `TestBackwardCompat::test_legacy_only_field_used_when_*` pins both paths.
- **The kill-switch's intent is now honored.** Before T3.4, the kill switch reverted ONLY the transport — the input gate stayed at the streaming-tuned ceiling. An operator killing streaming would still let 50K-token prompts through, hitting the same KF-002 layer 3 wall they were trying to avoid. After T3.4, killing streaming reverts BOTH the transport AND the input gate to legacy-safe defaults.
- **Per-provider legacy defaults** in the migration tool reflect KF-002 observed walls. The 24K openai / 36K anthropic numbers come from sprint.md and known-failures.md — they're not arbitrary. Operators who have field-observed thresholds different from these should hand-edit the migration output before merging.
- **Three-state precedence** is intuitive: CLI override → split field (transport-aware) → `max_input_tokens` (compat). `TestCliOverrideOrthogonal` pins that CLI always wins, including under kill-switch — operators retain per-call control regardless of the global config state.
- **Truthy/falsy matching mirrors `_streaming_disabled()`.** Same set of accepted values: `("1", "true", "yes", "on")` case-insensitive. `TestSplitBehavior::test_kill_switch_truthy_variants_each_revert` pins six variants; the falsy counterpart pins five.
- **Migration-tool idempotence** is critical for re-runs. The helper checks `"streaming_max_input_tokens" not in m` before injecting — re-running produces zero new report entries on an already-migrated config. Confirmed by direct invocation: first run produces 2 split_injected entries, second run produces 0.

## Known Limitations

1. **Schema fields are optional, not required.** Configs without the split fields continue to work via backward-compat. This is intentional — forcing the split would break every external fork's model-config.yaml. If future cycles want to force the split, add a deprecation warning when `max_input_tokens` is read and the split fields are absent.
2. **Per-provider legacy defaults are conservative.** The 24K openai / 36K anthropic numbers reflect KF-002 known-good values; operators with field-observed walls different from these should hand-tune before adopting. `--cycle103-split` migration tool reports what it injected so operators can audit before merging.
3. **No active deprecation of `max_input_tokens`.** The field stays in the schema indefinitely as the backward-compat path. If a future cycle wants to deprecate it, add a warning at lookup time when the split fields are absent.
4. **Migration tool extension is end-to-end-untestable without `ruamel.yaml`.** The current dev environment lacks `ruamel`, so the full migration flow couldn't be smoke-tested via CLI. The internal helper (`_apply_cycle103_split`) was directly invoked via `importlib` and confirmed correct (2 entries split, idempotent on second run). Operator deployments with `ruamel` installed get full coverage.
5. **Headless adapters (claude_headless / codex_headless / gemini_headless) not wired.** These adapters bypass the cheval HTTP path entirely; KF-002 layer 3 doesn't apply to them. Out of scope. If a future regression surfaces in the headless path, extend the gate.
6. **Bedrock adapter not wired.** Bedrock has its own retry/back-off semantics. Not in cycle scope. Sprint 2 corpus may surface signal here.
