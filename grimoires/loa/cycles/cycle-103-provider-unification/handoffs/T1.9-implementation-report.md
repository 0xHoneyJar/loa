### Why architectural closure is the right disposition

The ledger discipline (header of `known-failures.md`, lines 5-15) accepts
status transitions when:
1. The failing code path no longer exists, OR
2. The failure has been reproduced at ≥3 recurrences and an upstream
   issue closes it, OR
3. An empirical re-run produces a clean signal.

KF-008 hits condition (1): the four files that produced the failure
(`adapters/google.ts` plus the `call_api_with_retry` / raw-curl fallback
chain in flatline-*.sh) are deleted or routed through the cheval
substrate. The empirical re-run (condition 3) is operator-deployment
work — it requires live API budget and a real PR target, both outside
the cycle-103 sprint-1 implementation scope. T1.9 records the
architectural closure now; operator deployment confirms with live data.

## Technical Highlights

- **Transitive closure is the strongest disposition for "code-path retired" failures.** When the code that produced a defect has been deleted, "did it reproduce?" is unanswerable in the original sense — the original path doesn't exist. The substitute path (cheval `httpx`) was independently verified by T1.0 spike data. Together they constitute architectural closure.
- **KF-008's symptom was body-size-dependent.** The 3 original observations at 297KB / 302KB / 318KB request-body sizes, with anthropic+openai succeeding at <125KB on the same invocations, strongly suggested an undici / Node-fetch HTTP/1.1 keep-alive interaction with body size — exactly the kind of failure mode that disappears when the entire HTTP loop is replaced. The cycle-103 unification deletes the Node-side HTTP loop for provider calls.
- **Upstream issue #845 stays open as a vendor-history artifact.** The Loa-side issue is closed by the cycle-103 architectural change, but the underlying behavior (Node fetch SocketError mid-stream on ≥300KB to Google) may still affect other Node-side consumers. The issue is filed for community archaeology, not for vendor escalation. AC-1.6 says "Update upstream #845 if vendor-side reproduces" — the closure is route (a), not vendor-side, so no vendor escalation is needed.
